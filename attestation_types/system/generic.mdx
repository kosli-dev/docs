title: "System type: generic"
description: The generic system attestation type — record a free-form attestation with an explicit compliance status.
---

`system:generic` is for attesting facts that don't fit any other type. The attestation carries an explicit `is_compliant` flag (the caller decides) and accepts arbitrary user data via `user_data`. There's no scanner-specific tile renderer — the attestation appears in the generic JSON view.

Use this when you have a one-off fact to record and no system type covers your case. For org-specific evidence shapes you'll reuse, prefer a [custom attestation type](/administration/managing_custom_attestation_types/overview) instead.

## Versions

| Version | Status | Notes |
|---|---|---|
| `v1` | Current | Initial release. Requires `is_compliant: bool`. |

## What the attestation accepts

A generic attestation carries:

- **`is_compliant`** *(required)* — boolean. The caller decides whether the attestation is compliant. There's no schema-based check on findings; Kosli stores whatever the caller passed.
- **`user_data`** *(optional)* — arbitrary JSON. Surfaces in the attestation tile as a JSON block; useful for context the audit trail should preserve.

## Compliance behaviour

The system type's jq rule is simply `.is_compliant` — the value of the flag drives compliance. If the caller passes `is_compliant: false`, the attestation is non-compliant; `true` is compliant.

## How to use it

The data shape and compliance behaviour are identical to the legacy `kosli attest generic` command. Until system-type CLI flags are everywhere, the easiest path today is:

- Use [`kosli attest generic`](/client_reference/kosli_attest_generic) for everyday workflows
- Use the `system:generic` route when you specifically want a system-type-flavoured attestation (e.g. to standardise on one attestation pipeline across types)

Both paths produce equivalent attestations.
