---
title: "System type: Pull request"
description: The pull-request system attestation type — record that a change was made via a reviewed pull request.
---

`system:pull_request` is for attesting that a code change came in through a pull request on GitHub, GitLab, Bitbucket, or Azure DevOps. The Kosli CLI fetches pull-request data from your git provider's API and records the URL, state, author, approvers, and commits.

## Versions

| Version | Status | Notes |
|---|---|---|
| `v1` | Current | Initial release. Compliance is structural — pull-request data must be supplied. |

Pin a version with `@vN` (e.g. `system:pull_request@v1`); omit it to track latest. See the [versioning section](/attestation_types/system/overview#versioning).

## What the attestation accepts

A pull-request attestation carries:

- **`git_provider`** *(required)* — one of `github`, `gitlab`, `bitbucket`, `azure`
- **`pull_requests`** *(required)* — an array of pull requests associated with the change. Each entry includes:
  - `url` — the PR URL
  - `state` — open, merged, closed
  - `merge_commit` — the merge commit SHA (where applicable)
  - `approvers` — list of users who approved
  - `author`, `title`, `head_ref`, `created_at`, `merged_at` — PR metadata
  - `commits` — the list of commits the PR carries

## Compliance behaviour

Structural only: the attestation is compliant when pull-request data is supplied (the array can be empty). The contents — whether any PRs are present, whether they have approvers, what state they're in — don't affect compliance.

For compliance opinions like *at least one PR must exist* or *every PR must have an approver*, write a rego policy and record the verdict via [`kosli evaluate`](/tutorials/evaluate_trails_with_opa) — the OPA tutorial uses pull-request approval as its worked example.

## How to use it

The legacy `kosli attest pullrequest_*` commands (one per git provider, each hardcoding the "at least one PR" compliance rule) continue to work and remain the recommended path for now if you want that opinion baked in. See [`kosli attest pullrequest_github`](/client_reference/kosli_attest_pullrequest_github), [`kosli attest pullrequest_gitlab`](/client_reference/kosli_attest_pullrequest_gitlab), [`kosli attest pullrequest_bitbucket`](/client_reference/kosli_attest_pullrequest_bitbucket), and [`kosli attest pullrequest_azure`](/client_reference/kosli_attest_pullrequest_azure). The `system:pull_request` route gives you the same data shape and tile but without the built-in compliance opinion.
