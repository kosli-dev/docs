---
title: "System type: Pull request"
description: The pull-request system attestation type ‚Äî record that a change was made via a reviewed pull request.
---

`system:pull_request` is for attesting that a code change came in through a pull request on GitHub, GitLab, Bitbucket, or Azure DevOps. The Kosli CLI fetches pull-request data from your git provider's API and records the URL, state, author, approvers, and commits.

## Versions

| Version | Status | Notes |
|---|---|---|
| `v1` | Current | Initial release. Compliance is structural ‚Äî pull-request data must be supplied. |

Pin a version with `@vN` (e.g. `system:pull_request@v1`); omit it to track latest. See the [versioning section](/attestation_types/system/overview#versioning).

## What the attestation accepts

A pull-request attestation carries:

- **`git_provider`** *(required)* ‚Äî one of `github`, `gitlab`, `bitbucket`, `azure`
- **`pull_requests`** *(required)* ‚Äî an array of pull requests associated with the change. Each entry includes:
  - `url` ‚Äî the PR URL
  - `state` ‚Äî open, merged, closed
  - `merge_commit` ‚Äî the merge commit SHA (where applicable)
  - `approvers` ‚Äî list of users who approved
  - `author`, `title`, `head_ref`, `created_at`, `merged_at` ‚Äî PR metadata
  - `commits` ‚Äî the list of commits the PR carries

## Compliance behaviour

Structural only: the attestation is compliant when pull-request data is supplied (the array can be empty). The contents ‚Äî whether any PRs are present, whether they have approvers, what state they're in ‚Äî don't affect compliance.

For compliance opinions like *at least one PR must exist* or *every PR must have an approver*, write a rego policy and record the verdict via [`kosli evaluate`](/tutorials/evaluate_trails_with_opa) ‚Äî the OPA tutorial uses pull-request approval as its worked example.

## How to use it

```shell
kosli attest system pull_request \
  --flow my-flow \
  --trail $(git rev-parse HEAD) \
  --name pull-request \
  --git-provider github \
  --github-token "$GITHUB_TOKEN" \
  --github-org your-org \
  --repository your-repo
```

Pass the right credentials for your provider (`--github-token`, `--gitlab-token`, `--bitbucket-username` + `--bitbucket-password`, or `--azure-token`) and the CLI fetches the pull requests associated with the current commit, populating the `pull_requests` array.

To pin a version, suffix `@vN` on the type (e.g. `kosli attest system pull_request@v1`).

The legacy provider-specific commands ([`kosli attest pullrequest_github`](/client_reference/kosli_attest_pullrequest_github), [`kosli attest pullrequest_gitlab`](/client_reference/kosli_attest_pullrequest_gitlab), [`kosli attest pullrequest_bitbucket`](/client_reference/kosli_attest_pullrequest_bitbucket), [`kosli attest pullrequest_azure`](/client_reference/kosli_attest_pullrequest_azure)) are deprecated ‚Äî teams should migrate to `kosli attest system pull_request` with `--git-provider`.
