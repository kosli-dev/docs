`system:sonar` is for attesting the result of a SonarQube or SonarCloud analysis. The Kosli CLI fetches the analysis from your Sonar instance (or accepts a pre-fetched JSON) and records the quality-gate status, project details, and conditions.

## Versions

| Version | Status | Notes |
|---|---|---|
| `v1` | Current | Initial release. Compliance is structural — Sonar analysis results must be supplied. |

## What the attestation accepts

A Sonar attestation carries:

- **`sonar_results`** *(required)* — an object describing the analysis:
  - `analysedAt` — timestamp of the analysis
  - `revision` — git commit SHA the analysis was run against
  - `project` — project identifier (e.g. `{ key, url }`)
  - `qualityGate` — the quality-gate verdict, including `status`, `name`, and the list of evaluated `conditions`
  - `serverUrl`, `taskId`, `status` *(optional)* — additional Sonar metadata

## Compliance behaviour

Structural only: the attestation is compliant when Sonar analysis results are supplied. The quality-gate status is recorded as a fact but doesn't itself drive compliance.

For compliance opinions like *the quality gate must be OK*, write a rego policy and record the verdict via [`kosli evaluate`](/tutorials/evaluate_trails_with_opa).

The Sonar tile renders the quality-gate name, status, and each condition with its threshold and actual value — so reviewers can see exactly what the analysis returned.

## How to use it

The legacy `kosli attest sonar` command (which hardcodes the "quality gate must be OK" rule) continues to work and remains the recommended path for now if you want that opinion baked in — see [`kosli attest sonar`](/client_reference/kosli_attest_sonar). The `system:sonar` route gives you the same data shape and tile but without the built-in compliance opinion.
