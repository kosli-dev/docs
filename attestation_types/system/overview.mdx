System attestation types are Kosli-curated attestation types available to every organization. They ship with:

- A **schema** that validates the shape of the data you submit.
- A **default compliance rule** so the attestation has a sensible verdict out of the box.
- A **type-specific tile renderer** in the UI — e.g. a severity breakdown for security scan types — rather than a generic JSON dump.
- (Where applicable) An option to **attach a rego policy** that overrides the default compliance rule, letting you express your own standards without writing a new attestation type.

Unlike custom attestation types, system types are not authored per-organization — they're maintained by Kosli and shipped as part of the platform. Unlike generic attestations, they understand the shape of your data and can render it meaningfully.

## How a system type is structured

Each system type has:

- **A name** — used in the CLI command (e.g. `system:sarif`) and the type listing.
- **One or more versions** — older versions remain available so attestations made against `v1` keep validating even after `v2` ships with a different shape.
- **A schema (per version)** — defines what data the attestation accepts.
- **Compliance rules (per version)** — jq expressions evaluated server-side at attest time.
- **A tile template (per version)** — how the attestation renders in the Kosli UI.

## Compliance: structural by default, policy-driven on request

System types use the same two-layer compliance model across the board.

**Layer 1 — Structural check.** Every system type validates the submitted data against its schema. If the required fields are missing or malformed, the attestation is rejected. By default, this is the only compliance check Kosli applies — the attestation records that the data exists and is well-formed, treating it as a fact rather than imposing an opinion about whether the underlying findings are acceptable.

**Layer 2 — Optional rego policy.** Every system type also accepts an optional `evaluation_result` field — the output of `kosli evaluate` with a rego policy. When you supply it, the policy's `allow` flag drives compliance. When you don't, layer 1's structural check alone determines compliance.

This means you choose how opinionated Kosli is about your data:

- **Just facts** — submit the data, no policy. The attestation records what happened; compliance is structural.
- **Policy-driven** — submit the data alongside a rego evaluation result. Your rules decide what counts as compliant.

<Note>
`system:generic` is the one exception — it carries an explicit `is_compliant` flag in its data, and that flag is the default compliance signal (rather than just structural completeness). A rego policy, when attached, overrides the flag.
</Note>

The pattern is identical across types, so a customer who learns it for one system type can apply it to any other. [See SARIF for a worked example →](/attestation_types/system/sarif)

## How they relate to other attestation types

System types are the strategic direction for Kosli's built-in attestation types. Today some attestation types (`snyk`, `junit`, `jira`, `sonar`, `pull_request`) are dedicated built-ins with their own CLI commands. Over time, Kosli is moving these to system types so:

- Adding a new type doesn't require new CLI work — just a new YAML definition.
- Customers see a consistent CLI and tile experience across types.
- Versioning becomes explicit instead of implicit.

Existing dedicated commands (e.g. `kosli attest snyk`) continue to work; the migration to system-type equivalents is incremental.
