`system:junit` is for attesting test results parsed from JUnit-format XML. Most test runners (pytest, JUnit itself, Mocha with `mocha-junit-reporter`, Jest with `jest-junit`, Go's test runner via `go-junit-report`, and so on) produce JUnit XML; the Kosli CLI parses it into the structure this attestation accepts.

## Versions

| Version | Status | Notes |
|---|---|---|
| `v1` | Current | Initial release. Compliance is structural — JUnit results must be supplied. |

## What the attestation accepts

A JUnit attestation carries:

- **`junit_results`** *(required)* — an array of test-suite results. Each suite includes:
  - `name` — the suite name (typically the test class or file)
  - `failures` — count of test failures
  - `errors` — count of test errors
  - `skipped` — count of skipped tests
  - `total` — total tests
  - `duration` — wall-clock duration in seconds

The CLI extracts these fields from a directory of JUnit XML reports.

## Compliance behaviour

Structural only: the attestation is compliant when test-suite results are supplied. The contents of those results (whether any tests failed or errored) don't affect compliance.

For compliance opinions like *fail if any suite has failures or errors*, write a rego policy and record the verdict via [`kosli evaluate`](/tutorials/evaluate_trails_with_opa).

## How to use it

The legacy `kosli attest junit` command (which hardcodes the "zero failures, zero errors" compliance rule) continues to work and remains the recommended path for now if you want that opinion baked in — see [`kosli attest junit`](/client_reference/kosli_attest_junit). The `system:junit` route gives you the same data shape and tile but without the built-in compliance opinion.
