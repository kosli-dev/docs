`system:junit` is for attesting test results parsed from JUnit-format XML. Most test runners (pytest, JUnit itself, Mocha with `mocha-junit-reporter`, Jest with `jest-junit`, Go's test runner via `go-junit-report`, and so on) produce JUnit XML; the Kosli CLI parses it into the structure this attestation accepts.

## Versions

| Version | Status | Notes |
|---|---|---|
| `v1` | Current | Initial release. Compliance is structural ‚Äî JUnit results must be supplied. |

Pin a version with `--type-version v1` on the CLI or `type_version: v1` in a flow template; omit it to track latest. See the [versioning section](/attestation_types/system/overview#versioning).

## What the attestation accepts

A JUnit attestation carries:

- **`junit_results`** *(required)* ‚Äî an array of test-suite results. Each suite includes:
  - `name` ‚Äî the suite name (typically the test class or file)
  - `failures` ‚Äî count of test failures
  - `errors` ‚Äî count of test errors
  - `skipped` ‚Äî count of skipped tests
  - `total` ‚Äî total tests
  - `duration` ‚Äî wall-clock duration in seconds

The CLI extracts these fields from a directory of JUnit XML reports.

## Compliance behaviour

Schema-validated only: the attestation is compliant when JUnit results were supplied and parse correctly. The contents ‚Äî pass/fail counts per suite ‚Äî are recorded as facts but don't drive compliance.

For opinions on the content ‚Äî *fail if any suite has failures or errors*, *allow up to N failures*, *require a minimum number of suites to run* ‚Äî write a rego policy and record the verdict via [`kosli evaluate`](/tutorials/evaluate_trails_with_opa) as a separate attestation. See [Migrating from `kosli attest junit`](/tutorials/migrating_to_system_types) for a copy-paste rego policy that reproduces the legacy compliance rule.

## How to use it

```shell
kosli attest system junit \
  --flow my-flow \
  --trail $(git rev-parse HEAD) \
  --name unit-tests \
  --results-dir build/test-results/
```

Point `--results-dir` at the directory containing your JUnit XML reports. The CLI parses every XML file in the directory into the `junit_results` array.

To pin a version, add `--type-version v1` to the CLI command.

The legacy [`kosli attest junit`](/client_reference/kosli_attest_junit) command is deprecated ‚Äî teams should migrate to `kosli attest system junit`. See [Migrating from legacy commands](/tutorials/migrating_to_system_types) for the step-by-step, including a rego policy that reproduces the legacy compliance rule.
