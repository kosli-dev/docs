title: Custom attestation types
description: Custom attestation types let an organization define its own evidence shapes — schemas and compliance rules — when no system type fits.

A **custom attestation type** is one your organization defines, rather than one Kosli ships. You author the JSON Schema (optional) that describes the data shape and the [jq](https://stedolan.github.io/jq/) rules (optional) that determine compliance. At least one of the two is required.

Use a custom type when you have an evidence shape that's unique to your organization — internal QA reports, deployment manifests, license attestations, custom security tooling — and no [system type](/attestation_types/system/overview) covers your case.

## How a custom type works

A custom attestation type definition has three parts:

1. **A name** — used to reference the type when attesting (e.g. `custom:security-scan`).
2. **A JSON Schema** *(optional)* — defines what data the attestation accepts. Submissions that don't match the schema are rejected at attest time.
3. **A jq evaluator** *(optional)* — one or more jq expressions evaluated against the attestation data. All rules must return `true` for the attestation to be considered compliant.

At least one of (2) or (3) must be provided. A type with a schema but no jq rules validates structure but doesn't make a compliance claim; a type with jq rules but no schema makes a compliance claim against whatever data the caller submits.

### Example

A type that accepts a security scan summary and requires zero critical and fewer than five high vulnerabilities:

```hcl
resource "kosli_custom_attestation_type" "security_scan" {
  name        = "security-scan"
  description = "Validates security scan results"

  schema = jsonencode({
    type = "object"
    properties = {
      critical_vulnerabilities = { type = "integer" }
      high_vulnerabilities     = { type = "integer" }
      scan_date                = { type = "string" }
    }
    required = ["critical_vulnerabilities", "high_vulnerabilities", "scan_date"]
  })

  jq_rules = [
    ".critical_vulnerabilities == 0",
    ".high_vulnerabilities < 5"
  ]
}
```

## Versioning

Custom types are versioned. When you change a type's schema or jq rules, Kosli creates a new version automatically. Attestations made against `v1` continue to validate against `v1` even after `v2` is published — your history stays meaningful when policies tighten or shapes evolve.

## How custom types compare to system and generic

| | Custom | [System](/attestation_types/system/overview) | [Generic](/attestation_types/system/generic) |
|---|---|---|---|
| Defined by | Your organization | Kosli (shipped in code) | Per-attestation (just a flag) |
| Schema validation | Optional, you author it | Built-in per type | None |
| Compliance | Your jq rules | Built-in jq rules + (some types) optional rego | Caller passes `is_compliant` |
| Tile renderer | Generic JSON view | Type-specific (where built) | Generic JSON view |
| Cross-org reuse | No — per-org definition | Yes — every org sees the same types | Yes — universal |

## Creating and managing custom types

Custom attestation types can be managed via:

- **The Kosli Terraform provider** — recommended for production workloads, keeps your type definitions version-controlled alongside the rest of your infrastructure. See [Managing custom attestation types](/administration/managing_custom_attestation_types/overview) for the full guide.
- **The Kosli CLI** — `kosli create attestation-type` for quick creation and prototyping. See the [getting started attestations guide](/getting_started/attestations) for the basics.

## Reporting a custom attestation

Use `kosli attest custom` to report an attestation against a custom type. See [`kosli attest custom`](/client_reference/kosli_attest_custom) for the full flag reference.
